# 关键词库管理功能 - 设计方案

## 一、功能概述

### 核心能力
1. **手工管理** — 用户可增删改查关键词，并为每个词赋予标签
2. **AI理解** — 系统根据标签智能调整搜索、聚类、分析策略
3. **自动扩展** — AI从已采数据中发现相关热词，推荐给用户

### 业务价值
- **更精准的监控** — 产品名/竞品/吐槽点分类清晰，针对性采集
- **更智能的分析** — 卖点vs吐槽点自动识别，情感判断更准确
- **持续演进** — 词库自动学习网络热词，跟上舆情变化

## 二、标签体系设计

### 标签定义
| 标签 | 英文代码 | 用途 | 示例 |
|---|---|---|---|
| **产品名** | `product_name` | 监控主体，核心实体 | 星海手机、星海Pro |
| **口号** | `slogan` | 品牌记忆点，营销核心 | 星耀未来、科技向善 |
| **宣传语** | `promo` | 广告文案，营销话术 | 千元旗舰、性价比之王 |
| **吐槽点** | `complaint` | 负面关键词，用户痛点 | 发热、卡顿、售后差 |
| **卖点** | `selling_point` | 正面关键词，产品优势 | 续航长、性价比高、屏幕好 |
| **竞品** | `competitor` | 竞争对手，对比分析 | 小米、OPPO、vivo |
| **相关** | `related` | 泛相关词，扩展监控 | 手机壳、充电器、系统更新 |
| **近似** | `similar` | 同义词，语义扩展 | 续航≈电池、发热≈烫手 |

### 标签作用机制
```
采集阶段：
  product_name/competitor → 构造搜索关键词
  similar → 扩展搜索范围（"续航"搜索时也找"电池"）

分析阶段：
  complaint → 识别为负面信号，提升风险分
  selling_point → 识别为正面信号，降低风险分
  slogan/promo → 识别营销内容，标记为中性

聚类阶段：
  similar → 合并同义词簇（"发热"和"烫手"归一组）
  related → 关联分析（提到"充电器"时联想到产品）

检索阶段：
  用户问"电池问题"，自动召回标记为similar的"续航"相关帖
```

## 三、数据库设计

### 表结构

#### 1. keywords（关键词主表）
```sql
CREATE TABLE keywords (
    id INTEGER PRIMARY KEY,
    word TEXT NOT NULL,              -- 关键词
    tag TEXT NOT NULL,                -- 标签（product_name/complaint等）
    entity_id TEXT,                   -- 关联的监控实体（可选）
    weight REAL DEFAULT 1.0,          -- 权重（0-1，影响搜索/聚类优先级）
    source TEXT DEFAULT 'manual',     -- 来源（manual=人工/auto=AI发现）
    note TEXT,                        -- 备注
    created_at TEXT,
    updated_at TEXT,
    UNIQUE(word, tag, entity_id)      -- 同一词可有多个标签
);

CREATE INDEX idx_keywords_tag ON keywords(tag);
CREATE INDEX idx_keywords_entity ON keywords(entity_id);
CREATE INDEX idx_keywords_word ON keywords(word);
```

#### 2. keyword_suggestions（AI推荐词表）
```sql
CREATE TABLE keyword_suggestions (
    id INTEGER PRIMARY KEY,
    word TEXT NOT NULL,
    suggested_tag TEXT NOT NULL,      -- 建议的标签
    entity_id TEXT,
    score REAL,                       -- 推荐置信度（0-1）
    reason TEXT,                      -- 推荐理由（"与'发热'共现120次"）
    source_docs TEXT,                 -- 来源帖子doc_ids（JSON数组）
    status TEXT DEFAULT 'pending',    -- pending/approved/rejected
    suggested_at TEXT,
    reviewed_at TEXT,
    UNIQUE(word, suggested_tag, entity_id)
);

CREATE INDEX idx_suggestions_status ON keyword_suggestions(status);
CREATE INDEX idx_suggestions_entity ON keyword_suggestions(entity_id);
```

## 八、技术要点

### 1. 同义词扩展策略
```python
# 检索时扩展
query = "电池问题"
similar = keyword_mgr.get_similar_words("电池")  # [续航, 耐用]
expanded_query = " OR ".join([query] + [w['word'] for w in similar])
# → "电池问题 OR 续航 OR 耐用"
```

### 2. 权重影响
```python
# 权重决定扩展优先级
weight >= 0.8: 强关联，always扩展
0.5 <= weight < 0.8: 中关联，可选扩展
weight < 0.5: 弱关联，不扩展
```

### 3. AI发现去重
```python
# 避免重复推荐
existing = set(kw['word'] for kw in keyword_mgr.list())
suggestions = [w for w in discovered if w not in existing]
```

### 4. 标签优先级
```
分析时优先级: complaint > selling_point > slogan > related
采集时优先级: product_name > competitor > similar
```

## 十、风险与应对

### 风险1: AI推荐噪声
- **风险** — 发现的词不相关（如"手机"被推荐为complaint）
- **应对** — 设置阈值（frequency>10, score>0.75），人工审核

### 风险2: 标签冲突
- **风险** — 同一词被标记为complaint和selling_point
- **应对** — 允许多标签，分析时按上下文判断优先级

### 风险3: 词库膨胀
- **风险** — AI持续推荐，词库从50→5000，难以维护
- **应对** — 
  - 定期清理低权重(<0.3)且未使用的词
  - 设置词库上限（per entity: 500词）
  - 按最后使用时间淘汰
