# 舆情监控与分析产品设计方案（Youdoo Box 品牌）

> 立项与开发依据 · v1 版本 · 编制日期 2026-07-06
> 基于 yuqing 技术底座（Python，约 2000 行，5 次提交，已用微博/知乎真实数据跑通）
> 遵循四项已定产品战略：①内部自用为主、SaaS 仅预留 ②国内为主、出海二期 ③采集混合（核心买数据、长尾抓取）④v1 先服务执行岗分析师、数据质量是地基

---

## 〇、给决策者的一页纸（Executive Summary）

**一句话：** 我们已经有了一台"能分析舆情的发动机"，现在缺的是"把它装进一台能全公司天天开的车"——而车的地基是数据质量，不是新功能。

**最重要的判断（贯穿全文）：** yuqing 的算法能力已经**过剩**（采集→ABSA→风险分→P0/P1 预警→竞品 SOV→可溯源周报→诉求 backlog→事件时间线→AI 问答全链路都在）。从"工具"到"产品"的真实差距只有三处**地基**，全部与功能无关：

1. **数据质量**——已被真实测试坐实的三个硬伤：关键词串味（搜 Youdoo Box 串出 Doo Prime 外汇/创维新闻）、微博搜索无点赞/转发/粉丝数（影响力加权在代码里塌缩成常数）、黑猫投诉登录墙未通。**脏数据喂给高层看板，比没有产品更糟**——老板把"Doo Prime 外汇"当成自家舆情，会一次性摧毁对整个系统的信任。
2. **常驻可靠性**——现在是"一个分析师本机的登录态 Chrome + 手动跑 run.py"。搬到服务器无人值守 7×24 跑通，是"工具→产品"的真正分水岭，早于一切界面美化。
3. **组织责任（RACI）**——数据质量的"最后一公里"是人不是代码：词表要人持续收紧、账号池要人巡检过验证码、危机要人核实上报。**无 owner = 常驻后迟早集体静默失败**。

**v1 该做什么（一句话）：** 冻结新分析功能，把全部预算和工时投向"让现有能力在服务器上无人值守跑通 + 花小钱做第三方数据 POC 修数据源 + 把 must_not 串味过滤真正写进代码 + 复用现有看板加一个薄标注端点 + 把责任落成一张 RACI 表"。

**v1 明确不做（经评审裁掉的过度设计）：** 不迁 Postgres、不建 Vue SPA、不建整套 RBAC/身份域、不上 Redis/Airflow、不建 metrics_daily rollup、不养 Node 端 ECharts 渲染进程、不建企微/钉钉/邮件通道、不建结构化复盘知识库。**这些一律推到"第 2 个角色真正上线"时再建。**

## 一、产品定位与愿景

### 1.1 产品定位

**一句话价值主张：**
> 让 Youdoo Box 的每个岗位，都能在第一时间拿到**可信、可溯源、按角色裁剪**的全网口碑事实——执行岗把噪声变成事实，中层把事实变成战情，高层把战情变成决策。

**定位三要素：**

| 维度 | 定位 | 不做什么 |
|---|---|---|
| 服务对象 | 内部自用为主（一家出海硬件公司，<50 人量级） | 不提前为对外客户/多租户/计费投工程 |
| 平台范围 | 国内为主（微博/知乎/抖音/小红书/B站/贴吧/黑猫/电商评论） | 出海（YouTube/Amazon/Reddit/独立站）列入二期 |
| 产品本质 | 一套**单一事实源**（干净语料 + 风险分 + ABSA）× 多角色多视图 | 不是各角色各建一张表、各导各的 Excel |

### 1.2 愿景与演进

- **v1（现在）：** 一个**可信的数据地基** + 一个分析师愿意每天在里面干活的工作台。
- **中期：** 三层角色（执行岗工作台 / 中层战情室 / 高层一屏概览）共享同一事实源，靠视图和分发裁剪差异，而非新增采集。
- **远期（仅预留，不投工程）：** 核心表加一个默认为空的 `tenant_id` 列，品牌词/采集参数外置化——未来若真要对外 SaaS，加多租户无需重构核心表。**仅此，不为想象中的客户多写一行。**

### 1.3 一条设计红线（命门）

**上层每一个看板的可信度 = 执行岗数据质量的函数。** 串味或漏采会被逐级放大成错误决策。因此"数据完整度/可信度标记"和"doc_id 溯源"一样，是**全站一等公民**——宁可少一个花哨图表，也不能让高层把一张没标注的空数据当成"今天平安无事"。

## 二、目标用户与角色需求矩阵

三层价值链（自下而上）：**执行岗产出可信结构化事实（数据地基）→ 中层聚合成部门战情与周期报告（决策中间件）→ 高层拿一句话日报+危机叫醒做战略与风险决策（决策终点）。**

### 2.1 高层（CEO / CPO / COO）

| 角色 | JTBD（要做的决策） | 要什么 | 用哪些功能（复用 yuqing） | 成功指标 |
|---|---|---|---|---|
| **CEO** | 每天 30 秒确认"品牌今天是否安全"；危机时第一时间被叫醒且知严重度；季度看健康趋势 | 现在有没有正在烧的火？口碑变好还是变坏？我会不会最后一个知道？ | 一句话日报（`insights.oneliner`）+ P0 危机叫醒（`alerts`）+ 品牌健康趋势单页（风险分时间线）+ 一键下钻事件时间线 | 危机知晓 MTTA 分钟级；"被下属先口头告知"次数=0；**误报叫醒<1 次/季（零容忍）** |
| **CPO** | 把全网抱怨/建议转成有优先级的产品需求；验证固件/新品口碑；决定哪个诉求进 roadmap | Top3 产品痛点按体量×情绪怎么排？新版本比上一代好还是差？ | 诉求→需求 backlog（人在环路）+ ABSA 六维口碑（硬件/系统/游戏兼容/售后/价格/物流）+ 版本前后口碑对比 | backlog 被采纳进 roadmap 的条数；重大产品决策引用舆情数据的比例 |
| **COO** | 掌握部门响应效率；把握竞品 KINHANK 整体态势；把控采集与 AI 成本 | SOV 此消彼长？团队处置够不够快？系统稳不稳、这月花多少钱？ | 竞品 SOV（`report.sov`）+ 部门战情概览 + 成本配额熔断仪表（`budget.py`，暴露到 /overview）+ 健康三态 | SOV 季度变化方向；危机平均处置时长下降；可用性达标 + 成本零击穿配额 |

> **高层排期结论（解决评审口径冲突）：** 高层功能价值最高但最脆弱——一次误报叫醒就失去信任。因此**高层功能必须建立在下游可信之上**。v1 只交付**确定性的"一句话日报"**（`oneliner` 无 LLM、已有雏形，飞书推送即可，风险低）；**"P0 危机叫醒"作为高价值单点在阶段3上线，前置条件是中层数据稳定 + 误报治理机制（30min 人工核实门）成型**。二者不是同一件事，不要混为"高层 v1 全上"。

### 2.2 中层（运营 / 营销 / 产品总监）

> 中层视图的本质是执行岗事实源的**"换聚合口径"，不新增采集**。工程量小、见效快。

| 角色 | JTBD | 要什么 | 用哪些功能 | 成功指标 |
|---|---|---|---|---|
| **运营总监** | 每周向上汇报运营战况；识别需运营介入的负面/热点；给团队派活 | 本周热点与上升话题？哪些负面必须回应、优先级？ | 可溯源周报（`report.build_report`+`validate_citations`）+ 上升话题（`analytics.rising_topics`）+ P1 预警队列 | 周报按时交付率；负面 48h 响应率；话题在登上热搜前被提前识别的比例 |
| **营销总监** | 评估 campaign 传播效果；监测品牌声量/情绪；识别 KOL | 活动打响没有？声量涨没涨、情绪正负？竞品在营销上做了什么？ | 活动效果评估（时间窗前后对比，v1 弱版）+ 情绪趋势 + 竞品营销事件时间线 | 可量化复盘的活动占比；声量目标达成率。**诚实 gap：微博无转发/粉丝→传播深度指标不准，需第三方数据补齐** |
| **产品总监** | 盯产品线各维度口碑；对标 KINHANK；把结构化反馈交产品团队 | 我哪个维度口碑掉了、为什么？竞品哪点比我强、哪点被吐槽？ | ABSA 维度趋势 + 竞品对标矩阵 + 与 CPO 共享诉求 backlog | 受关注维度改善幅度；对标结论落地为迭代项的数量 |

> **中层排期：** 先上**运营总监**（周报已现成，改造成本最低）；再上**产品总监**（复用 ABSA+竞品对标）；**营销总监最后上**（强依赖传播互动数据，须等第三方数据接入）。

### 2.3 执行岗 7 类（v1 主战场）

| 岗 | JTBD | 用哪些功能 | 成功指标 & 诚实 gap |
|---|---|---|---|
| **③舆情监控（v1 核心角色）** | 实时盯全网、发现负面/危机并分级 | P0/P1 分级预警 + 事件簇冷却 + 关键词管理 + 健康熔断 + 只读看板 | 危机发现时效；误报/漏报率；**关键词精度（串味过滤后相关率）——整个产品数据地基第一指标** |
| **①用户意见建议收集** | 系统化收集建议/抱怨、结构化去重归类喂产品 | backlog 录入 + 人在环路标注工作台 + ABSA + CLI backlog | 有效诉求条数/期；人工修正后误标率下降。gap：黑猫供不出投诉源 |
| **⑤竞品监控** | 持续跟踪 KINHANK 动态/口碑/SOV | 竞品 SOV + 竞品事件时间线 + 竞品 ABSA + **竞品负面不误告警** | 竞品动作发现时效；SOV 报告交付率 |
| **④品牌传播分析** | 定期解读口碑走势与正负构成、产出洞察 | 风险分/情绪趋势 + 可溯源周报 + AI 问答（RAG-lite）+ 事件时间线 | **报告溯源率=100%**；洞察被采纳数；单份分析耗时下降 |
| **②传播数据收集** | 汇总各平台声量/互动做传播报表 | 跨平台采集 + 传播数据导出 + 平台分布视图 | 覆盖平台数；采集成功率；**数据缺口被显式标注不假装完整**。gap：微博无互动、知乎无 id、黑猫未通 |
| **⑥活动效果评估** | 每次 campaign 做声量/情绪/传播复盘 | 活动打点（时间窗）+ 前后对比 + 异常检测 | 每次活动有量化复盘。gap：**缺转化/销量数据，归因只能停在声量-情绪层，算不出真 ROI** |
| **⑦UGC 信息收集** | 收集优质 UGC 供营销二次复用 | UGC 采集 + 分类 + 情绪/质量标注 + 导出 | 优质 UGC 产出量；可复用率。gap：小红书/电商评论强依赖登录态，覆盖有洞 |

**执行岗排期结论：** v1（P0）只做前 4 类**（舆情监控 + 意见收集 + 竞品监控 + 品牌分析）**——它们直接生产数据地基，底座已能支撑，边际成本最低。**核心投入不是加功能，而是数据质量治理。** 后 3 类（传播/活动/UGC）排 v1.5，因它们强依赖互动数据，须等第三方数据接入。

## 三、核心使用流程

### 3.1 五类工作的端到端主链

```
全网信息汇总 → 整理分类 → 监控跟踪 → 解读汇报 → 深度分析
   (采集接入)   (质检+打标)  (预警)    (报告分发)  (对标/时间线/诉求)
```

具体节点（自动/人工标注）：

1. **采集调度**［自动 / `run.py`］→ 按 `watch.yaml` 混合抓取（核心平台第三方数据、长尾 opencli 登录态）→ `store` 增量水位去重 → 产出 raw + 三态健康信号
2. **数据质检闸门**［人工 / 舆情监控岗，**v1 新增强制环**］→ 抽检机器分类、判串味、剔竞品品类噪声 → 回写 clean 打"已核"标记 → 产出干净语料 + 串味词表（回喂 `watch.yaml`）。**脏数据在此拦截**
3. **分析打标**［自动 / Haiku/Sonnet 分层路由］→ 极性/ABSA/投诉/Bug/功能诉求 + 证据校验 → 产出 features（带 evidence 可溯源）
4. **风险评分**［自动 / 线性加权］→ 影响力×极性×危机信号；**微博缺互动时退化为"存在即计"并打降级标记**
5. **分发**［自动 + 分角色］→ P0/P1 走实时飞书卡片（窄通道、事件簇冷却），其余沉淀周报（数字注入+引用校验，每条溯源到原帖）→ 切三份视图（执行岗明细 / 中层周期 / 高层概览）
6. **决策/行动**［人工 / 各角色］→ 回流到诉求池 / 危机 SOP / 活动评估三条闭环

### 3.2 场景走查 A：一次危机从苗头到复盘（Youdoo Box 固件发热退款）

| 时点 | 动作 | 谁 | 系统能力 |
|---|---|---|---|
| Day0 | 微博出现"发热退款"苗头 | — | 采集命中 + `alerts.evaluate` 出 P0 |
| Day0+15min | 飞书卡片推送舆情岗 + 值班人 | 舆情监控岗 | **P0 触达 SLA ≤15min** |
| Day0+30min | **人工核实**：判真伪/是否串味/是否竞品锅 → 确认为真 | 舆情监控岗 | **30min 人工核实门（硬约束）**；timeline 显示"微博起火" |
| Day1 | 知乎深扒→事件簇累积，总监知会 CPO | 运营总监 | `content_cluster` 冷却去重、事件时间线 |
| Day2 | 抖音扩散→**高层叫醒**，定统一口径+固件热修+客服话术 | CEO/CPO | 分级上报，高层知会 ≤1h |
| Day3 | 声量回落→关闭预警 | 舆情岗 | 稳健 z-score 异常回落 |
| Day5 | 复盘卡归档扩散链+处置+效果，"发热"进需求池 | 产品岗 | v1 落飞书文档（非结构化知识库） |

### 3.3 场景走查 B：分析师的一天（v1 主用户）

- **09:00** 打开工作台，先看**顶部三态健康条**——任一平台 suspect/fail 亮红条+平台名，"先信任数据源再看内容"。
- **09:10** 处理**复核队列**（默认过滤：机器最没把握的——低置信/反讽/规则与 Claude 分歧/命中 must_not 边缘）；抽屉逐条快捷键打标（1正/2中/3负/i反讽/s水军/r串味），写回 review 表。"清空队列"是当日 KPI，不是无限流。
- **10:00** 发现"Youdoo Box"又串出"Doo Prime 外汇"→一键把该关键词命中样本标脏 → 到**关键词管理**页收紧 must_not（旁边显示该词近 7 天命中量/被标脏率）。
- **14:00** AI 问答："发热问题主要在哪些平台"→ 附来源 doc_id + "依据 N 条·可能不全"脚注。
- **17:00** 导出当周诉求 CSV → 人在环路勾选确认 → 推飞书多维表给产品总监。

## 四、产品功能地图

按 5 类工作模块化，标注**复用 yuqing** vs **新建**，分 v1 / 二期。

### 模块① 全网信息汇总（采集接入层）
| 能力 | 状态 | 分层 |
|---|---|---|
| 微博/知乎登录态采集（`collect.py` opencli 桥） | 复用（已验证） | v1 |
| 混合接入：核心平台第三方数据服务 → 统一归一化 CleanDoc | **新建 vendor 字段映射适配器**（复用 `normalize`/`_to_int`） | v1 |
| 自有店电商评论官方 API（淘宝 TOP/京东宙斯） | **新建**（唯一能合规拿官方 API 的场景） | v1 |
| 增量水位 + 幂等去重（`watermark` + `UNIQUE(platform,native_id)`） | 复用 | v1 |
| 抖音/小红书/B站/贴吧登录态真实跑通 | 声明但未验真，需实测 | 二期 |

### 模块② 整理分类【v1 核心 · 数据质量地基】
| 能力 | 状态 | 分层 |
|---|---|---|
| **must_not 串味过滤真正接进 analyze** | **净新建**（今天 must_not 仅是 watch.yaml 空字段，无任何 .py 引用） | **v1 最高杠杆** |
| 情绪极性/强度/置信（规则词典 + Claude 双路径） | 复用（`analyze.rule_extract`/`claude_extract`） | v1 |
| ABSA 六方面（价格/质量/性能/服务/物流/系统） | 复用（`ASPECT_LEXICON`+`aspect_breakdown`） | v1 |
| 防幻觉护栏（evidence 必须正文逐字子串） | 复用（`_validate_evidence`） | v1 |
| **人工标注/复核队列**（复用 review 表 schema，加轻流程） | **半新建**（表在、流程/UI 无） | v1 |
| 分层模型路由省钱 + 成本配额熔断 | 复用（`route_model`+`budget.py`） | v1 |
| 话题聚类：Claude/规则 topic_label 字符串归并 | 复用（**非真语义聚类**） | v1 |
| 真嵌入语义聚类 + SimHash 洗稿去重 | 需向量库 | 二期 |

### 模块③ 监控跟踪
预警 P0/P1（`alerts._level`）· 事件簇冷却（`COOLDOWN`+`content_cluster`）· 竞品不误告警（`self_entities` 过滤）· 数据健康预警 · 负面放量异常（中位数+MAD 稳健 z-score）· 上升话题 — **全部复用**。
**新建 v1：** 常驻定时调度（APScheduler/cron 替代手动 run.py）+ **deadman 心跳自监控**。
飞书推送复用；企微/钉钉/邮件 adapter **不建（YAGNI，没人要）**。

### 模块④ 解读汇报
周期报告（数字注入+LLM 三段散文+引用校验）· 可溯源到原帖 · 老板一句话日报 · AI 问答 RAG-lite（SQL LIKE 检索，**无向量库**）· 只读看板（stdlib http.server）· CLI — **全部复用**。

### 模块⑤ 深度分析
竞品 SOV+NSR（复用）· 跨平台事件时间线（复用，**时间+关键词近似，非真传播树**）· 诉求聚合 backlog（复用，**人在环路只出 CSV 不自动建工单**）· 竞品方面级对比（新建，组合 `aspect_breakdown×entity`）· 活动效果评估（v1 用 rising_topics 前后窗拼弱版）。
真传播关系图谱 + 事件级实体 + 活动打标/UTM → **二期**。

## 五、前端设计

### 5.1 v1 前端的核心裁决（吸收评审）

> **评审冲突：** frontend 视角主张 v1 就建 Vue3 SPA 并压 70% 预算；roadmap 主张冻结新功能、用 Metabase。**裁定：roadmap 对。** 无设计师、疑似单 dev，2-3 周搭出完整 SPA（虚拟滚动+抽屉+关键词 diff 回滚+双栏对比）不现实，会挤占真正该做的数据地基工时。

**v1 前端方案（最小可用）：**
- **分析师工作台**：复用现有 stdlib `dashboard.py` 只读页 + **一个薄写入端点**（POST 回写 review 表）+ CSV 勾选回写。这就能让分析师打标、走复核队列、收紧关键词。`// ponytail: 薄端点够用，Vue SPA 等第 2 个分析师上线再建`
- **中层/高层看板**：先用现有周报/日报/看板；**需要图表时直接上 Metabase 接库**（图表/权限/分享 BI 已解决，不自建）。
- **高层一句话日报 + 危机卡片**：复用 `oneliner` + 飞书卡片，同源。
- **stdlib 看板保留**：降级为内部健康/debug 页，零成本继续服役。

### 5.2 分析师工作台关键组件（v1）
- **顶部三态健康条**（复用 `health.banner`）：任一平台 suspect/fail 红条+平台名——"没抓到≠没负面"。
- **实时信息流表格**：列=平台/时间/作者/摘要/情绪/风险分/方面标签/原帖链接；每行带 doc_id + 原帖外链。
- **一键打标 + 复核队列**：默认过滤"机器最没把握的"；批量选中批量打标；"串味"一键标脏并回流关键词治理。
- **关键词管理**：以 `watch.yaml` 为 single source of truth，表单编辑 entity/aliases/must_not；每个词旁显示近 7 天命中量/被标脏率，串味词红标。
- **导出**：当前筛选结果一键 CSV/xlsx（复用 `backlog_csv`）。

### 5.3 中高层视图（v1 用 Metabase / 现有输出，SPA 二期）
- 高层一屏四格：品牌健康分（大数字+环比箭头）/ 危机灯（绿黄红）/ 三条自然语言结论（`oneliner`）/ 竞品对标一行。**危机灯是唯一强交互**，其余一眼概览。移动优先（与飞书卡片同源）。
- 中层三部门差异化默认视图：运营看情绪趋势+活动、营销看 SOV+传播、产品看方面口碑+诉求。

### 5.4 数据可视化规范（吸收 dataviz + 评审）
- 图表类型定型：情绪趋势=堆叠面积/折线；SOV=100% 堆叠条/环形；方面口碑=方面×情绪热力图；Top 负面=排行列表+分数条（不硬做成图表）。
- **无障碍配色**：情绪不靠红绿单维（避色盲误读），用 **蓝=正/灰=中/橙红=负** + 文字标签+形状冗余，对比度守 WCAG AA。
- **数据完整度是一等公民**：凡涉加权的图表统一显式打"仅计数/未加权"角标（微博降级数据），**绝不用视觉精度掩盖数据缺口**。
- 交互克制：默认 hover 看值、可点下钻到支撑帖，不加动画/3D。

### 5.5 前端技术选型
- **v1：** stdlib http.server（保留）+ 薄写入端点 + Metabase（中高层图表）。**不上 Vue/React SPA、不上 SSR、不养 Node 端 ECharts→PNG 渲染进程（评审判定过度设计）。**
- **二期（第 2 个角色上线后）：** Vue3 + Vite + Element Plus + ECharts（B 端数据密集后台，表格/表单/抽屉开箱，中文生态成熟）；明确不上 SSR（内部登录后应用无 SEO）。图表以 ECharts option 为单一契约，需报告出图时再考虑服务端渲染。

## 六、后端与数据架构

### 6.1 现有底座是资产，别推倒
doc_id 确定性契约、`UNIQUE(platform,native_id)` 幂等、raw 层 append-only、clean⋈features 稳定读模型、run_log 三态健康、watermark 增量、usage 成本配额——**这些是产品化最难补的部分**。真正要改的只有三件：并发写、身份与可见范围、采集常驻化。

### 6.2 v1 存储裁决（吸收评审的直接冲突）

> **评审冲突：** backend 主张"第一期就迁 Postgres，越早越省"（因身份表依赖并发写）；roadmap 主张"只在阶段2多用户并发才迁，MVP 用 SQLite+WAL 顶住"。**裁定：roadmap 对。v1 只服务单个分析师，单写者 SQLite+WAL 完全够；迁库是唯一破坏性改动，是过早优化。**

- **v1：** SQLite + WAL（开 WAL 让常驻 cron 写 + 少量读不锁冲突）。
- **阶段2（多用户真出现）：** SQLite→Postgres 15，JSONB 承接 raw payload，`INSERT OR IGNORE`→`ON CONFLICT DO NOTHING`。schema 高度可移植，是机械活。届时引入 Alembic 管迁移。
- **不上：** 对象存储（S3/MinIO）、时序库（InfluxDB/Timescale）、metrics_daily rollup、extractor_version 版本回填——**115 条/期直接查 clean⋈features 毫秒返回，全是数据量到百万行才需要的机制（评审判定预建）**。留清晰升级路径即可。

### 6.3 服务拆分：只拆一条真接缝

> **原则：** 有状态/单并发/重副作用的才拆独立进程；纯函数/批处理留库模块。**7 个"服务"里只有采集该物理独立**（它耦合登录 Chrome、单并发、失败模式独特，不能拖垮在线服务）。analyze/score/report/alert 都是同库纯函数，拆成 RPC 只增延迟和故障面、零收益。

- **v1：** 全部单进程，采集/分析仍由调度器 import 调用。
- **阶段2：** 抽出**采集 worker** 这一条真接缝。**不上 RQ+Redis/Airflow/Celery/K8s**（评审判定：115 条/期量级下 Redis 队列是多余组件，APScheduler/cron 单机足够）。

### 6.4 权限：v1 不建，靠约定

> **评审判定过度设计：** backend 第一期整套身份域（users/roles/departments/api_tokens/entity_acl/audit_log）+ RBAC WHERE 过滤——在只有 1 个分析师用时建全套权限是典型 YAGNI。

- **v1：** 看板加反代 basic auth 起步（内部<50 人，三行配置）。角色差异化靠**分发内容切三份视图 + 飞书群分层**，不做账号权限系统。
- **阶段3（全公司铺开、第 2+ 角色真上线）：** 才建最小可用的"角色×可见实体集"WHERE 级过滤 + session 登录 + audit_log。**不自建 OAuth、不做行级 policy 引擎、不接企业 SSO**（除非二期对外 SaaS）。

### 6.5 API 与演进
- **v1：** 现有 stdlib http.server + 一个薄写入端点（回写 review）。
- **阶段2+：** FastAPI 单应用（自动 OpenAPI，Web/飞书/移动共用读模型），端点围绕读模型：`/overview`、`/entities/{id}/timeline`、`/entities/{id}/sov`、`/alerts`、`/reports/{run_id}`、`/backlog`、`/ask`、`/docs/{doc_id}`（溯源）。写路径不开放为公开 API，只暴露标注/诉求确认这类受限 analyst 写操作。

### 6.6 调度与可观测（v1 必须补的保命组件）
- **调度：** APScheduler 进程内 或系统 cron（run.py 注释里已埋此计划），分平台错峰（高频词 15-30min、常规 2-4h、长尾日级）。
- **幂等已是强项，别动：** doc_id + `ON CONFLICT IGNORE` + watermark + `clean_missing_features` 重入，整条链天然可重放。
- **⭐ deadman 心跳自监控（评审 high-value，提到 v1）：** 采集 worker 每轮更新 `last_run_ts`，超阈值未更新就飞书告警"系统自己哑了"。**否则常驻后调度器整个哑掉，静默失败会伪装成"今天没舆情"，高层以为平安无事——几十行的事却是无人值守的保命组件。**
- 复用 run_log 做运行看板一页；补结构化日志（JSON 行）+ `/healthz`。**不上 full tracing**（量级不值）。

## 七、数据采集基建（混合方案落地）

### 7.1 纠正一个错误心智模型（评审 reality-check，最关键）

> **"买官方 API 补互动数"是伪命题。** 国内微博/知乎/小红书/抖音**均无面向第三方的开放内容搜索 API**。按"接口对接"排期会严重 mis-scope。真正可"买"的是**第三方舆情/垂类数据服务**（识微/清博/新榜/千瓜/飞瓜/蝉妈妈/魔镜）的数据导出。**唯一能名正言顺买官方 API 的是自有店电商评论**（淘宝 TOP/京东宙斯，拿自家产品评论，合规、ROI 高、法律风险低——应明确进 v1）。真实工作是 vendor 商务采购 + 写字段映射适配器，不是接 SDK。

### 7.2 国内平台"买 vs 抓"矩阵

| 平台 | 官方 API | 第三方数据 | 登录态抓取 | **建议** |
|---|---|---|---|---|
| **微博** | 对第三方基本关闭 | 识微/清博/鹰眼覆盖舆情 | 已跑通但**无赞/转发/粉丝** | **买第三方舆情**（危机首发地，缺互动数使风险分失真，需补齐+合规兜底） |
| **知乎** | 无第三方内容 API | 覆盖弱 | 已跑通，无 id 从 url 派生 | **继续抓**（量小、第三方性价比低，接受影响力字段退化） |
| **抖音** | 内容搜索不开放通用第三方 | 飞瓜/蝉妈妈（付费）+巨量算数（趋势） | 风控极强、视频为主难做文本 | **买第三方**（自建 ROI 最差、风控最狠，最该外包） |
| **小红书** | 无开放内容 API | 千瓜/灰豚（付费） | 风控严、需频繁验证 | **买第三方（千瓜）**（消费电子口碑主战场，值得付费） |
| **B站** | 无正式但有半公开 web 接口 | 火烧云/新站 | 风控相对宽松、评论弹幕结构化好 | **继续抓**（性价比最高的自抓平台） |
| **贴吧** | 无 | 弱 | 可行但内容杂、串味重 | **长尾抓取、低优先**（v1 只挂词监控不深抓） |
| **黑猫投诉** | 无 | 部分舆情服务商已聚合 | **登录墙未通（硬事实）** | **优先买第三方聚合拿到**（若买识微/清博通常已含黑猫）；否则维持登录态桥重试 |
| **电商评论** | 淘宝 TOP/京东宙斯**（自家店可合规拿）** | 魔镜/知瓜（跨店品类） | 天猫风控+滑块严 | **自有店走官方 API（进 v1）+ 竞品 KINHANK 走第三方或轻抓** |
| **新闻门户** | 无通用搜索 API | 舆情服务商全覆盖；聚合数据 API 极便宜 | 多数可直接 HTTP 抓 | **第三方已含 或 聚合 API 补**（成本极低、优先级可低） |

### 7.3 账号池运维（当前 0 基建、自抓能否常驻的命门）
- **专用可牺牲号**：每自抓平台备 3-5 个专用监控号，与公司真实运营号**完全隔离**（监控号被封是预期损耗，绝不用官方号抓取）。
- **养号**：新号 2-4 周养号期再投采集，冷启动直接高频搜索必被风控。
- **账号池状态表**：`platform, session, 状态[健康/验证中/已封], 最后成功时间, 当日请求数`——**复用已有 run_log/health 三态落地，不新建监控系统**。触发验证码即标"验证中"切下一号，人工过验证后回池。
- **风控规避**：随机间隔 3-8s、单号日配额、错峰、住宅代理 IP 与账号 1:1 稳定绑定、UA/指纹固定。
- **现实**：小红书/抖音号封得最快（周级损耗），知乎/B站存活久。

### 7.4 采集服务器架构（常驻化——评审 reality-check：这是净新建+持续运维，不是配置项）

> **"搬到服务器 cron 跑"背后是一整套从 0 到 1 的运营：登录态服务器持久化、账号池、代理 IP、接码平台。这被 acquisition 自己承认是常驻化"第一道坎"，却被 7 份设计降级成 gap——本方案把它提为 v1 关键路径交付物。**

- **常驻调度**：一台常驻 Linux 小机（2C4G 够）跑 APScheduler/cron，分平台错峰。
- **⭐ 登录态服务器持久化（v1 关键路径，非 gap）**：opencli browser 持久 profile + Xvfb 跑 headful/headless；**session 失效自动飞书告警重登**——这是全部无人值守/常驻/多用户的真正关键路径。
- **去重/水位/断点续跑**：复用现有 UNIQUE 约束、watermark、单平台失败不阻塞其他平台；补调度级重试（指数退避）。
- **健康自检 + 心跳**：复用 health 三态 + "抓到但 0 条有效 id 判 suspect"字段映射失败保护 + budget 熔断；补"每平台连续 N 次 fail 自动告警飞书叫醒运维 owner"+ deadman 心跳。

### 7.5 v1 采集侧总账（评审 gap：把分散成本合成一个盘交决策者拍板）

| 项 | 量级（国内 2026 行情区间） |
|---|---|
| 第三方舆情服务（识微/清博级，含微博互动量+黑猫+新闻） | 2-8 万元/年 |
| 垂类数据（千瓜小红书 或 飞瓜抖音，先只买最痛 1 个） | 1-3 万元/年 |
| 自有店电商官方 API | 基本免费（已是商家） |
| 聚合新闻 API | 千元级/年 |
| 采集服务器（2C4G 云主机） | 2-4 千元/年 |
| 账号池运维（住宅代理+接码+备号） | 3-5 千元/年 |
| **一次性工程**（第三方 API 集成+搭常驻服务） | 2-4 人周 |
| **持续人力**（采集运维 owner） | 0.1-0.2 FTE |
| **v1 第一年采集侧总盘** | **约 6-15 万元 + 0.2 FTE**（数据服务占大头） |

相比自建全平台爬虫团队（人力+封号损耗+法律风险），买第三方数据在国内是更省的选择——这是"混合"策略成立的经济基础。

### 7.6 合规红线
- **PIPL**：采集含昵称/头像/发言=个人信息，公开信息用于舆情监测走正当利益+最小必要；**不采手机号/身份等敏感信息**（store 已只存昵称+粉丝数，保持此边界）。
- **平台 ToS**：主流平台均禁未授权自动化抓取——这正是用专用可牺牲号、控频率、买第三方兜底的根本原因，把违约风险控制在"损失几个监控号"而非"公司主体被诉"。
- **刑事红线**：不破解平台加密/签名（黑猫是让真实登录浏览器自己渲染，**没有逆向破解，这个边界要守住**）；不抓私域数据对外。
- **第三方数据合规转嫁**：买正规 vendor 的数据，合规责任部分转移给持牌服务商——这是"买"相对"抓"的合规价值。
- **需法务背书**：PIPL 正当利益基础、各平台 ToS 风险敞口、数据留存期限（建议 12 个月窗口）、境内存储——当前是"工程判断"，缺法务红线确认。

## 八、完整业务逻辑链条与跨部门工作流

### 8.1 信息主链 + 四道人在环路闸门

不重写主链，在既有节点间插入四个"人在环路"闸门，分发端按角色切三份视图：

1. **数据质检质检**［每日 / 舆情监控岗］：抽风险分+随机分层，高风险全检、低风险抽样入队 → 逐条判极性/ABSA/串味 → 回写 features 打"人工已核" → 串味词进 watch.yaml。**SLA：当日出数据前完成高风险全检。**
2. **危机响应 SOP**［分级上报］：见 8.2。
3. **诉求→需求闭环**［每周 / 意见收集岗→产品岗］：`insights.backlog` 按（类型×话题）聚类排序 → 人工去重剔串味判真需求 → 一键确认推**飞书多维表**（当前终点，无研发工单对接）。**硬约束：绝不自动灌工单。**
4. **竞品情报流转**［周期 / 竞品监控岗］：竞品 SOV/方面口碑/负面（不叫醒自家）→ 人工解读意图对标 → 汇入竞品报告。仅竞品异常放量走 P1 提示，不占危机通道。

### 8.2 危机响应 SOP（分级上报 + 30min 人工核实门）

> **评审 high-value（应被所有视角采纳为硬约束）：P0 危机"先推送 + 30min 人工核实门"。** 串味/竞品误告警会一次性摧毁高层对系统的信任，这个人工闸门比任何算法都值钱。

```
alerts.evaluate 命中 P0/P1 → 事件簇冷却去重 → 实时飞书卡片（标注"待确认"）
   ↓ ≤15min 触达舆情岗+值班人
人工核实（≤30min）：判真伪/串味/是否竞品锅 → 真则升级、噪声则标抑制回写质检
   ↓
分级上报：P0 → 舆情岗→营销/运营总监→CEO/CPO 危机群"叫醒"（附一句话定性+建议口径）；P1 当日跟进
   ↓
处置+复盘：公关/产品/客服认领 → 事件闭合归档苗头时间线+处置+效果（v1 落飞书文档）
```
**SLA：** P0 触达 ≤15min、人工核实 ≤30min、高层知会 ≤1h；P1 当日闭环。冷却期内同簇不重复叫醒。

### 8.3 活动效果评估闭环
活动前（人工圈关键词/时间窗 + 自动拉基线声量/情绪/SOV）→ 活动中（rising_topics + 稳健 z-score 实时监测，跑偏可叫停）→ 活动后（前后差值+话题结构变化，**人工归因**）。
**硬约束：因果归因必须人工，系统只给前后对比数字不宣称因果**（声量变化可能来自大盘/串味/自然波动）。**gap：缺转化/销量，算不出真 ROI。**

### 8.4 组织 RACI 责任表（评审 gap：无 owner 比技术更致命）

> 词表 owner、采集运维 owner、危机 on-call、标注分析师工时——各视角零散点名，本方案汇总成一张表。**数据质量最后一公里是人不是代码，无 owner = 常驻后迟早集体静默失败。**

| 职责 | Owner | 工时占比 | 谁兼任 | 频率 |
|---|---|---|---|---|
| **词表/品牌词维护**（收紧 must_not、白/黑名单） | 运营/品牌方 | 0.05-0.1 FTE | 可由舆情监控岗兼 | 持续（每周收紧） |
| **采集运维**（账号池巡检、过验证码、补号、代理续费、vendor 对接） | 采集运维 | **0.1-0.2 FTE** | 可由数据分析师兼 | 每日巡检 |
| **数据标注/复核**（质检队列、纠偏） | 舆情监控岗 | 0.2-0.3 FTE | 分析师 | 每日 |
| **危机 on-call**（P0 核实、分级上报） | 值班分析师 | 排班 | 舆情岗轮值 | 7×24 排班 |
| **系统运维**（部署/DB/告警/备份） | dev | 0.1-0.2 FTE | 现有 dev | 按需 |

**注意：** 质检/危机核实是单点，值班人休假即断链——需明确 backup。

## 九、我们目前欠缺的条件与技术（按严重度）

### 🔴 致命（决定 v1 成败）
1. **串味过滤 0% 落地（最关键）**：grep 全仓 must_not 只出现在 watch.yaml 一个空数组，**无任何 .py 引用**。过滤器、复核队列、实体消歧全是净新建。Youdoo Box→Doo Prime/创维 的串味今天在代码里**完全没有任何拦截**。这是被全体反复引用却 0 行代码的地基。
2. **登录态无服务器持久化、无账号池**：单人单 Chrome，cookie 过期即断供，黑猫登录墙未通。这是全部无人值守/常驻的真正关键路径。
3. **组织无 owner**：词表/采集运维/危机 on-call/标注工时全无人认领。无 owner=常驻后集体静默失败。
4. **微博互动数据先天缺失**：score.py 的 `_influence(likes,comments,reposts,followers)` 四项全 0 时**塌缩为常数**，weibo 的 risk 实际=极性×危机信号。任何建立在微博自抓上的 SOV/影响力/传播深度**在第三方数据补齐前都是失真的**——无法自抓补齐，只能买第三方。
5. **合规缺法务背书**：PIPL 正当利益基础、ToS 风险敞口、数据留存期限均未经法务确认。

### 🟠 高（决定能否推广）
6. **并发安全存储缺失**：SQLite 单写者是多用户硬天花板（但 v1 单分析师用不到，阶段2 再迁）。
7. **认证与权限体系从零**：看板零鉴权，无角色无审计（阶段3 才需要）。
8. **常驻调度 + deadman 心跳缺失**：现手动跑，无"跑挂自动告警"。
9. **第三方 vendor 未做 POC**：识微/清博/千瓜/飞瓜的实际覆盖度/字段/时效/报价均未验证——不能拍脑袋把"买数据"写进架构。
10. **巴士因子=1**：2000 行单人所写、缺部署文档与关键路径测试，作者离开即死。

### 🟡 中（可带病运行、逐步补）
11. **标注→模型回喂未闭合**：v1 是"没有复利的人工跑步机"，分析师永远手动收紧关键词（回喂推 Phase2）——需量化工时预期。
12. **黑猫登录墙未通**：诉求/投诉闭环的"投诉源"在买到含黑猫的第三方数据前是空的。
13. **无向量库**：AI 问答/话题聚类停留在 SQL LIKE 与 topic_label 字符串，语义召回弱。
14. **无真 SimHash 洗稿去重**：只能挡完全复制粘贴，改写搬运漏网。
15. **时间戳水位只认 ISO**：微博/知乎"3 小时前"等相对时间不参与水位→退化为重抓（正确性 OK，量大后成本上升）。
16. **活动归因无转化/销量数据**：只能到声量-情绪层，算不出真 ROI。
17. **无 CI/备份恢复/发布回滚**：从"一个人的脚本"到常驻产品的工程基线空白。

## 十一、风险与优先级建议

### 11.1 Top 6 优先级（按杠杆排序，直接吸收评审）

1. **让现有能力在一台常驻服务器上无人值守 7 天跑通**：登录态服务器持久化+失效自动重登告警 + APScheduler/cron 错峰 + deadman 心跳。**这是"工具→产品"的真正分水岭，早于一切界面美化。**
2. **把 must_not 真正接进 analyze** + "疑似串味"复核队列（复用 review 表）+ 每个关键词显示近 7 天命中量/被标脏率。**被全体反复引用却 0 行代码的地基，杠杆最高。**
3. **第三方数据源付费 POC 先行再定架构**：优先一个含微博互动量+黑猫的全网舆情服务（识微/清博）+ 小红书垂类（千瓜）；自有店电商走官方 API。**修影响力退化和黑猫两个源头硬伤。**
4. **v1 不建 Vue SPA**：复用 stdlib dashboard/CLI + 薄写入端点做标注工作台；中高层用现有周报/日报，需图表才上 Metabase。**把省下的工时投数据层。**
5. **可信度贯穿**：数据完整度标记 + doc_id 溯源覆盖到日报/周报/看板，微博降级数据显式标注；**P0 危机"先推送+30min 人工核实门"防误叫醒高层。信任是这个产品的命门。**
6. **组织认账**：把词表 owner、采集运维 owner（0.2FTE）、危机 on-call、标注工时落成 RACI 表。**Postgres/RBAC/SPA 一律推到第 2 个角色真上线时再建。**

### 11.2 最大风险（致命级）

| 风险 | 影响 | 缓解 |
|---|---|---|
| **R1 串味摧毁信任** | 高层看到"Doo Prime 外汇"当自家舆情，对整个产品失去信任 | 词表收窄 + 给高层的内容**强制过人工审核门，宁漏不误报** |
| **R2 合规/封号** | 长尾登录态抓取违反 ToS，账号封禁即采集断供 | 核心走合规第三方数据，抓取仅长尾且低频+随机延时+专用可牺牲号 |
| **R3 无 owner 集体静默失败** | 账号被封→health 判 fail→**但没人处理**，再好的熔断也只是报警不修复 | RACI 表明确 owner + deadman 心跳 + 连续 N 次 fail 飞书叫人 |
| **R4 数据质量最后一公里是人不是技术**（认知性） | 没有持续标注就退化，这条**无法靠再写代码解决** | 组织上认账 + 量化标注工时 + Phase2 才闭合回喂降工作量 |
| **R5 巴士因子=1** | 作者离开即死 | 补部署文档 + 采集/评分/告警关键路径最小测试 |

### 11.3 核心取舍（一句话）

- **数据质量 > 新功能**：底座功能已过剩，瓶颈在源头不在算法——**冻结新分析功能，全投数据质量**。
- **无人值守可靠 > 界面美化**：常驻是"产品"与"工具"的分水岭，没跑通前看板再美也是单机脚本。
- **约定 > 权限系统**：v1 靠分发切视图+飞书群分层，RBAC 推到第 2 个角色上线。
- **买第三方数据 > 自建全平台爬虫**：这是"混合"策略成立的经济与合规基础，但**买之前先付费 POC，不拍脑袋写进关键路径**。
- **人工闸门 > 全自动**：串味判定、P0 核实、诉求进池、活动归因四处必须留人；采集/清洗/评分/报告/引用校验/去重冷却全自动。

> **一句话收尾：** 我们不缺一个能分析舆情的引擎，缺的是让它可信地、无人值守地、按角色地为全公司天天服务的地基工程。v1 把钱和工时押在数据质量、常驻可靠、组织责任这三块地基上——**先把"可信"做出来，再谈盖楼。**
