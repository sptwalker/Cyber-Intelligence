## 四、操作手册（怎么用）

启动看板：`python -m yuqing.dashboard yuqing.db`（当前跑在 http://127.0.0.1:8088）。首页导航有全部入口。

### 1. 配监控对象（搜什么）→ `/watch`
- 直接编辑 watch.yaml（种子词/别名/否定词/平台）。保存前强校验、自动备份 `.bak`、下轮生效。
- **别名写全**是提召回的第一杠杆（含常见错拼、中英文、简称）。

### 2. 登录 + 采集 → `/login`
- 看各平台登录态（微博/知乎等靠 opencli 复用你本机 Chrome 登录）。未登录点"打开登录页"人工扫码。
- 点「▶ 运行采集」跑一轮（实时进度 + 可停止）。

### 3. 训练：标注代表样本 → `/annotate`（核心）
- 系统按"最没把握 + 高影响 + 多样"挑样本给你标（不用你大海捞针）。
- 每条选**主体/立场/重要性**、鼠标划选**圈关键词**并指定角色。
- **圈词自动反哺**：判别词进关键词库待审、产品名进种子建议——**标注即同时提精度和扩召回**。
- 建议每个立场档（赞扬/中立/批评/吐槽/投诉/纯传播）至少标 2 条。

### 4. 让分类器生效
- 标注后，下次跑批的新帖自动用 few-shot 打多维标签。
- 想让**已有数据**也用上新分类器：清 features 重跑（`DELETE FROM features` + `python -m yuqing.run`，会有 LLM 费用）。
- 看效果：首页显示主体/立场分布；`python -m yuqing.cli classify eval` 算留出集准确率。

### 5. 官方账号白名单 → `/accounts`
- 登记官方/准官方/媒体账号 → 分析时**确定性判主体**（盖过 LLM，最准的一维）。

### 6. 种子增长（扩召回）→ `/watch` 的「🌱 种子建议区」
- 点"生成建议"挖候选词；系统把产品词列这里、评价词分流到 `/keywords`。
- 点"✓加入aliases"→ 一键写回 watch.yaml（保注释、备份），下轮多搜这些词。

### 7. 上线日常模式 → `/config`
- 运行模式选 `daily`（跑批推飞书）或 `training`（安静迭代不推）。
- 挂 cron 自动跑（见下）。

## 五、运维说明

### 定时自动跑批（外部 cron，零额外进程）
- **Windows 任务计划程序**：程序 `python`，参数 `-m yuqing.run`，起始位置=仓库根，触发器按需（如每小时）。
- **Linux crontab**：`7 * * * * cd /path/to/repo && python -m yuqing.run >> run.log 2>&1`。
- **存活告警**：再挂一个每小时的 `python -m yuqing.scheduler deadman`——长时间无成功跑批→飞书告警。

### 前提与注意
- 采集依赖 **opencli 保持登录态**（会话失效→采集 fail，系统会告警提示重登，不会把"没抓到"当"没负面"）。
- 跑批会调 LLM（DeepSeek/MiniMax，按量付费）；`YUQING_MAX_CALLS/TOKENS` 是每日熔断护栏。
- **跑批与看板尽量别同时大量写库**：`busy_timeout` 扛偶发，但两个重写者对撞会超时（本次调试踩过，故日常走 cron 单飞）。

### 数据与密钥
- `yuqing.db`（数据）、`yuqing_config.json`（密钥）、`*.log/*.bak` 均已 gitignore，不进仓库。
- ⚠️ 遗留：`docs/apiKey-6025102.csv` 是密钥文件（已 gitignore），建议确认是否需轮换并移出仓库目录。

## 七、当前系统能力全景

**采集**：watch.yaml 种子 → opencli 复用登录态搜 10 平台 → 相关性过滤（含别名 ∧ 排否定词，可选语义救回）→ 健康三态（"没抓到"≠"没负面"）。
**分析**：DeepSeek+MiniMax 交叉复核情感/方面 → **多维分类(主体×立场×重要性)** → 风险打分 → crisis_boost 每实体危机词。
**学习闭环**：主动学习挑样本 → 多维标注+圈词 → few-shot 范例 + 官方白名单 + 词分流反哺（判别词/种子词）。
**汇报**：数字代码注入 + LLM 只写措辞 + 引用校验（防幻觉）→ 报告/飞书。
**看板**（http://127.0.0.1:8088）：首页 / 登录采集 / 标注 / 监控配置 / 账号白名单 / 关键词库 / 战情室 / 高管概览 / 报告 / 系统配置。
**运维**：cron 定时 + deadman 存活告警 + 成本熔断 + 训练/日常模式切换。

## 八、已知局限与后续方向（诚实清单）

1. **立场维需要标注喂养**：目前只标了 5 条、覆盖 2 档立场，"吐槽/投诉/批评/中立"缺范例，靠零样本判、偏弱。**采到负面数据后补标这几档**才会准。
2. **主体维非白名单账号**：未登记的一律"用户·KOL"，准官方/未登记媒体识别不了——靠持续维护 `/accounts` 白名单补齐。
3. **粉丝数缺失**：opencli 对小红书/B站取不到粉丝数，"影响力/重要性"实际按点赞/声量排，非粉丝影响力。要名副其实需在采集层补粉丝提取。
4. **登录态非全自动**：微博/知乎/黑猫登录必人工过风控，做不到永不落地全自动；现实预期是无人值守 + 每周人工重登 1-2 次。
5. **Phase B 知识图谱（词-标签共现可视化）**：评审判定可选、YAGNI，暂缓。**仅当分类需要可解释性、或想可视化"哪些词组合→什么属性"时再做**——当前 few-shot 已满足核心需求。
6. **单机上限**：SQLite + stdlib 看板适合单机单人；多用户/高可靠需迁 Postgres + 正式 Web 框架（设计已预留升级路径）。
