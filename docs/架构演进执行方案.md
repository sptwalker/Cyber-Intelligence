# 核心架构加固执行方案

> 变更：`harden-core-architecture`  
> 目标：在不推倒现有模块化单体的前提下，修正数据归属、观测历史、搜索闭环、分析血缘与告警确认流程。

## 一、范围与原则

- 保留 Python 模块化单体、SQLite + WAL、外部 cron/systemd 调度。
- 所有数据库变更必须向后兼容：旧库首次打开时自动建新表、补列并回填关系。
- 不在本阶段引入微服务、Redis、消息队列或独立向量库。
- 先修正会导致统计错误的数据契约，再扩展多人协作能力。

## 二、目标数据模型

```text
documents(clean) ──1:N── raw_observations
       │
       ├──1:N── engagement_snapshots
       │
       ├──N:M── document_entities ──N:1── watch entities
       │
       └──1:N── analysis_results（按 analysis_version 留历史）

alerts/incidents：pending_confirmation → confirmed/escalated
                                      └→ suppressed/resolved
```

`clean.entity_id` 暂时保留为兼容字段；新的实体过滤、SOV 和告警以
`document_entities` 为准。待所有读取路径迁移完成后，再考虑移除兼容字段。

## 三、实施任务

- [x] 建立 `document_entities`，支持同一帖子归属多个自有/竞品实体。
- [x] 建立 `raw_observations` 和 `engagement_snapshots`，保留每次采集及互动量变化。
- [x] 所有 aliases 都参与平台搜索，并记录 `source_query`。
- [x] 建立 `analysis_results`，记录分析版本、引擎、模型、Prompt 版本和时间。
- [x] 建立 `incidents` 状态机，P0/P1 默认进入待人工确认队列。
- [x] 统一写接口的认证判断，补充依赖清单与核心回归测试。

## 四、验收条件

1. 同一 `(platform, native_id)` 可同时计入两个实体，且 SOV 不受采集顺序影响。
2. 同一帖子二次采集后，原始观测和互动快照均增加，当前互动量更新为较新值。
3. `aliases[1:]` 会实际发起搜索，而不只是参与相关性过滤。
4. 每条当前 feature 可追溯到一个 `analysis_version`，历史结果不被覆盖丢失。
5. P0 在人工确认前不会进入高层升级通道，抑制/确认/关闭均有审计记录。
6. 离线自检、调度自检和新增架构回归测试全部通过。

## 五、上线注意

- 首次运行会把旧 `clean.entity_id` 自动回填到 `document_entities`，不会删除旧字段或旧数据。
- 历史 `features` 没有 `analysis_version`，首次跑批会按 `opinion-v2` 重新分析一次；上线前应切
  `training` 模式并确认 LLM 每日预算，或暂时不配置 LLM key 让规则路径完成回填。
- 每个实体的唯一 aliases 现在都会实际搜索，请根据平台风控能力控制词数和调度频率。
- 历史采集无法补造 `raw_observations`/`engagement_snapshots`；这两类历史从升级后的首次采集开始累计。
