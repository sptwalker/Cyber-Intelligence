# 搜索-分析学习闭环 · 设计方案

> 目标：把当前"种子词采集 → 一次性分析"的单向管线，升级为**边用边学**的闭环——
> 训练模式人工标注 → 系统学出①更准的多维分类器②词-标签知识③更全的搜索种子 →
> 日常模式全自动跑、越用越聪明。本文是实现前的完整设计（本阶段不写实现代码）。

制定方式：3 个 agent 逆向映射现有代码钩子 → 4 个 agent 分组件设计 → 1 个 agent 对抗式评审，
本文已把评审挑出的问题全部消化进各阶段（见 §6 风险台账）。

---

## 0. 现状基线（可复用的半个闭环）

系统已有闭环的大半个雏形，本方案是"串起来 + 补缺口"，不是推倒重来：

| 你的设想 | 现有零件 | 状态 |
|---|---|---|
| 种子词 + 属性 | `watch.yaml`（aliases + type）+ `/watch` 网页编辑器 | ✅ |
| 训练模式挑代表样本 | `store.review_queue`（未复核 ∩ 机器最没把握）| 有雏形，缺多样性采样 |
| 人工标注落库 | `review` 表 + `store.add_review` | 有单维，缺多维/圈词 |
| 系统更新词库 | `keywords` 库（8 标签 + 自定义）+ `keyword_suggestions`（approve/reject）| ✅，缺"从标注反哺" |
| 重要词进种子库 | `analytics.suggest_targets`（只出建议，人在环路）| 有建议，缺"确认写回 watch" |
| 分类器 | `analyze` 的 polarity/risk + `cross_extract` 交叉复核 | 有粗粒度，缺多维 |
| 知识图谱 | 无 | 全新（且建议最后做） |

**关键事实**：`keyword_suggestions` 表已建，但**全仓无任何生产写入者**（一直空）——本方案的标注/挖词正好补上它。

## 1. 四条核心概念校正（不贯彻会踩坑）

**① 两类"词"方向相反，必须分流**
- **搜索词/种子**（watch aliases）→ 管**召回**（去平台搜什么）。加词=多搜。
- **特征/判别词**（关键词库）→ 管**精度**（搜回来怎么判）。如"翻车/官方回应/避雷"。
- 学到的词要分流：产品专名类 → 建议进搜索种子；立场/情绪类 → 留作判别特征。
- **盲目把高频词塞进搜索词会"串味"**——"游戏/手机"这种泛词让召回爆炸全是噪声。故种子增长坚持**半自动+人工确认**（决策②）。

**② 8 类标签 = 三个维度，不是扁平 8 分类**（决策①已定"多维"）
- **主体**：官方 / 准官方 / 媒体 / 用户·KOL
- **立场**：赞扬 / 中立 / 批评 / 吐槽 / 投诉 / 纯传播
- **重要性**：复用已有 `risk` / `mention_equiv`（见 §5-C 纠偏）
- 多维可独立训练/标注，组合表达 4×6 细分，且与现有 polarity/risk/KOL 天然衔接。

**③ 知识图谱是"解释器+特征源+建议器"，不是分类器**
- 光靠共现分类会失准（"退款/客服"在"用户投诉"和"官方处理"都出现）。
- 图谱负责：词↔标签关联强度（NPMI）、给分类器当特征、给标注挑词、给人看解释。
- **真正判类交给 LLM few-shot**（决策③）。两者分工，别让共现图当分类器。

**④ "代表样本" = 主动学习采样，不是随机列**
- 挑最值得标的：不确定的（低置信/两模型分歧）+ 多样的（embedding 聚类去近重）+ 高影响的（KOL）。
- 这三个信号系统全部现成——这是最省人力的一环。

## 2. 闭环全景

```
                          ┌──────────────── 日常模式（自动）────────────────┐
 watch.yaml 种子词 ──采集──→ clean ──分析──→ features(polarity/risk/主体/立场/重要性) ──→ 报告/飞书
      ▲                        │                    │
      │ D:一键确认写回          │ 低置信样本回流      │
      │                        ▼                    ▼
   种子建议 ◀──分流──┐   ┌─ 训练队列(主动学习采样) ─┐
   (产品词)          │   │  A:人工多维标注 + 圈关键词 │
                     │   └───────────┬───────────────┘
   判别词建议 ◀──────┘               │ 产出
   (立场词→keyword库)                ▼
                          annotations 表（doc×主体×立场×重要性×圈词）
                                     │
                    ┌────────────────┼────────────────┐
                    ▼                ▼                ▼
              C:few-shot 范例   B:词-标签图谱(可选)   A:圈词→词库建议
              (多维分类器)      (解释+特征+挑词)      (approve→分析生效)
```

一句话：**种子→召回→主动学习挑样本→多维标注+圈词→更新①分类器范例②词库③(可选)图谱④种子建议→人工确认扩种子→下一轮召回更全、打标更准。**

## 4. 推荐实施顺序（已按评审修正）

> 评审结论：原稿把"知识图谱(B)"排在分类器(C)之前是错的——B 与 C 从同一张标注表学习，
> 但 B 只输出"软提示"，是冗余的第二学习器。正确顺序让 **C 先落地产生价值，B 最后且可选**。

**Phase 0 · 修路由 BUG ✅（已完成）**
> `dashboard.py` do_POST 的 `/api/keywords` 分支曾被误嵌进 `/api/watch` 块，导致关键词增删/审核全 404。
> 这是整个闭环"圈词→建议→approve 进词库"的地基。**本次已修复并验证**（add/delete/approve 均 200）。

**Phase A · 标注控制台（训练模式）** ← 先做，一切地基
**Phase C · 多维分类器（LLM few-shot）** ← 真正的价值交付
**Phase E · 日常模式（自动化 + 回流）**
**Phase D · 半自动种子增长** ← 让"标注也能增强召回"（Phase 0 后任意时点可插）
**Phase B · 词-标签知识图谱** ← 最后、可选，仅当 C 被证明不够用时

### Phase A · 标注控制台 + 主动学习

**目标**：训练模式列出"最值得标的代表样本"，人工打三维标签 + 圈关键词，落库供后续学习。

**新增数据模型**（`store.py` 的 `_SCHEMA`，`CREATE TABLE IF NOT EXISTS` 零迁移）：
```sql
CREATE TABLE annotations (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  doc_id TEXT NOT NULL,
  subject TEXT,        -- 官方/准官方/媒体/用户·KOL
  stance TEXT,         -- 赞扬/中立/批评/吐槽/投诉/纯传播
  importance TEXT,     -- 高/中/低
  picked_words TEXT,   -- JSON: [{"word","role","span":[s,e]}]  role 取自 keywords.TAGS
  note TEXT, annotator TEXT DEFAULT 'local', sample_source TEXT,
  entity_id TEXT, ts TEXT );
```
> ★评审要求：**Phase A 必须先冻结此 schema + 把 SUBJECTS/STANCES 定为跨组件唯一常量**
> （放 `classify.py` 顶部，B/C/E 全部 import 同一份），否则枚举串味（"用户·KOL" vs "用户/KOL"）会让准确率评估虚假归零。全仓统一用 **`用户·KOL`**（中点）一种写法。

**主动学习采样**（新增 `analytics.active_sample`，复用 `review_queue` 口径 + `embed.cluster`）：
```
base = 0.35·不确定性(1-confidence) + 0.25·两模型分歧 + 0.20·KOL(log粉丝) + 0.20·风险
→ 取 base 高的 pool → embedding 聚类去近重(每簇取代表) → 返回前 20，每条附"采样原因"标签
```
> ★评审修正（采样↔范例冲突）：`risk` 对非负面帖恒 0，纯 base 排序会让队列长期**负面主导**，
> "赞扬/纯传播"正面帖几乎标不到 → Phase C 分层范例缺正面档。**对策**：队列保留
> **~30% 名额给"立场覆盖"**——按机器 polarity=pos/neu 且未标的帖补进来，保证 6 档立场都有样本可标。

**标注 UI**（新页 `/annotate`，与 `/keywords` 并列）：样本卡（正文可鼠标划选圈词）+ 三维单选（重要性用 `mention_equiv` 预填可改）+ 圈词角色下拉（复用 `keywords.TAGS`）+ 保存并下一条。

**闭环产出**：保存时每个圈选词 → `km.add_suggestion(word, role, entity_id, source_docs=[doc_id])` 进 `keyword_suggestions` 待审 → `/keywords` 页 approve → 进正式词库 → 下轮 `analyze.rule_extract` 读 complaint/selling_point 即生效。
> ★补召回缺口（评审 missing #3）：圈词 role='product_name' 的，**额外**进"种子建议"（seed_alias，见 Phase D），
> 让人工标注**既提精度又扩召回**——否则标注努力对召回零贡献，愿景落空一半。

**接线点**：`store.py`（+annotations 表、`add_annotation/annotation_candidates/annotated_count/latest_annotation`）；
`analytics.py`（+`active_sample`）；`dashboard.py`（+`render_annotate`、GET `/annotate`+`/api/annotate/queue`、POST `/api/annotate` 过 `_write_allowed`）；`keywords.py` 不改（复用 add_suggestion）。

**降级**：无 embedding key → 采样退化为"不确定性+KOL 排序"（不聚类），不崩；空队列显示"无待标样本"。

**验收**：内存 Store 塞 5 帖（含近重 2 条）→ 断言 active_sample 把近重折叠成 1 且高 base 优先；标注后该 doc 出队；圈词经 add_suggestion→approve 后进词库。

### Phase C · 多维分类器（LLM few-shot）

**目标**：在 `analyze_pending` 现有逐帖循环里**加一步引擎无关的分类**，判**主体×立场**两维（重要性派生，不烧 LLM），结果落 `features.signals`（零迁移）。

**新增 `yuqing/classify.py`**：SUBJECTS/STANCES 常量、`CLASSIFY_TOOL` schema、`build_exemplars`、`classify_llm`、`classify_rule`（离线兜底）、`validate_keywords`、`importance_bucket`、`evaluate`。

**范例检索** `build_exemplars(store, holdout, per_stance=2, cap=12)`：从 `annotations ⋈ clean` 取已标样本，**按 stance 分层取 medoid**（最中心的典型样例）喂 prompt；无 embedding key → 退化为每档取最新 N 条。
> ★评审修正：生产调用必须 `holdout=False`（默认 True 会永久划走 20% 标注只用于评估，生产白扔 1/5）。

**Prompt**：静态 system 讲清 taxonomy + 反讽提示；user 段放"范例(已人工标注)"+待标注帖；输出 `{subject{value,conf}, stance{value,conf}, keywords[]}`，keywords 必须逐字来自正文（防幻觉，`validate_keywords` 逐词 `in text` 过滤）。provider 复用 cross_extract 口径（deepseek→minimax）。

**接进 `analyze_pending`**（4 处，与 crisis_boost 覆盖同层）：
> ★评审修正（`day` 作用域）：分类调用 `guard(store, day,...)` 里的 `day` 只在 `if engine!="rule":` 块内定义——
> classify 调用必须**夹在该 if 块内**（L287-293 之间），否则块外调用 `day` 未定义 → NameError 打崩整个分析。
- A 批量分类（if 块内，try 吞异常→cls={}）；B 逐帖合并进 `sig["subject/stance/..."]`；C 重要性派生；D provider 小工具。
- **features 表 DDL 不动**，全落 signals JSON 键（下游 `json.loads(signals).get(...)` 现成范式）。需 SQL 过滤时再照 `store.__init__` 的 ALTER 迁移补列。

**评估** `evaluate`（CLI `classify eval`）：留出 20% 算各维 accuracy + 混淆矩阵（dict 计数打文本表），标注<10 条返回"标注不足"不报错。

**降级铁律**（任一层挂都不阻塞跑批）：无 LLM→`classify_rule` 粗推（stance 从 polarity+is_complaint+crisis）；无标注→跳过 LLM 走 rule；LLM 残缺/超预算→逐条丢弃走 rule。**主体维**优先查账号白名单（§4-Account，确定性），白名单未命中再由 LLM/followers 兜底——"纯传播"离线仍难识别，早期如实承认（见 §6）。

**接线点**：新文件 `classify.py`；`store.py::load_annotations`（新）；`analyze.py::analyze_pending`（4 处）；复用 `llm.chat_json`/`embed`/`score.mention_equiv`/`budget.guard`；`cli.py` 加 `classify eval`。

### §4-Account · 官方账号白名单（决策⑥，服务主体维）

**问题**：主体维的"官方/准官方/媒体"离线无数据源（`clean` 只有 `author`/`author_followers`，无认证/机构标记），纯 LLM 也只能靠语气猜。用一张**人工维护的账号→主体类型**白名单做**确定性判定**，命中即定，未命中才交给 LLM/followers 兜底。

**新表 `account_registry`**（`store.py` 的 `_SCHEMA`，零迁移）：
```sql
CREATE TABLE account_registry (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  platform TEXT,             -- 平台（空=跨平台匹配）
  author TEXT NOT NULL,      -- 账号昵称/handle（与 clean.author 对齐）
  subject_type TEXT NOT NULL,-- 官方/准官方/媒体
  entity_id TEXT,            -- 归属实体（空=全局，如通用媒体号）
  note TEXT, ts TEXT,
  UNIQUE(platform, author) );
```
> `用户·KOL` 不入表（默认兜底类），只登记非用户的三类，表小、维护成本低。

**判定优先级**（`classify` 内 `resolve_subject(row)`）：
1. `account_registry` 命中 `(platform, author)` → 直接返回其 `subject_type`（确定性，最高优先）。
2. 未命中 → LLM few-shot 的 subject 判定（有 key 且有范例时）。
3. 再兜底 → `author_followers` 粗启发（沿用 `analytics.kol_ranking` 口径），默认 `用户·KOL`。

**维护入口**（复用现有半自动闭环，不新造审核台）：
- **人工**：新增 `/accounts` 页（或挂在 `/keywords` 加一个 tab）增删账号→类型，`POST /api/accounts` 过 `_write_allowed`。
- **半自动**：标注时若某帖 subject 标为官方/媒体，**建议**把其 `author` 入库（进一个待确认列表，人工一键确认写 `account_registry`）——让标注顺带积累白名单。

**边界**：跨平台同名号靠 `platform` 区分；`author` 被平台改名会漏判（best-effort，靠人工补）；表空时全靠 LLM/followers，不崩。

**接线点**：`store.py`（+`account_registry` 表、`account_type(platform,author)` 查询、`add_account`）；`classify.py::resolve_subject`（判定优先级）；`dashboard.py`（`/accounts` 页 + `POST /api/accounts`）；标注保存时可选建议入库。

### Phase E · 日常模式（自动化 + 回流）

**关键框架判断**：训练/日常**不是两条管线**，是同一条 `collect→analyze→features` 之上的"调度策略 + 前台 UI + 是否推飞书"差异，用一个 config 开关切换，**数据模型不分叉**。

- **自动调度（决策⑦）**：**外部 cron 调 `python -m yuqing.run`**（`run.py` 现成 CLI 入口，零新代码、最可靠、进程退出不影响）。
  Windows 用任务计划程序、Linux 用 crontab（如 `*/60 * * * *`）。**不做进程内定时线程**（单机场景 cron 严格更省）。
  > `mode` 开关只影响 `run.main` 内部行为（是否推飞书、是否注入分类器），不负责"定时"——定时全交给 cron。
- **分类器接入**：`config.classifier_enabled` 为真时，`analyze` 注入 few-shot 范例；范例源优先 `annotations`，无则从 `review` 表自举；范例<2 条**不注入**（避免坏范例带偏，退化为现有 zero-shot）。
- **低置信回流（持续学习）**：`review_queue` 本就是活查询——每轮新产生的低置信帖，下次打开训练队列**自动出现**，零额外存储。仅新增 `store.training_queue` 加一层 embedding 多样性去重。
- **模式切换**：`config.mode ∈ {training, daily}`。training 不推飞书、前台弹"待标注 N 条"；daily 推飞书、队列静默累积。**两种模式都由同一个 cron 触发同一个 `run.main`**，差异只在 `run.main` 内部按 `mode` 分支（不再有"自动开火"之说——定时全交给 cron）。
  > ★评审修正：`config.save` 是白名单键驱动，且 `/config` 渲染是文本框——加 `mode/classifier_enabled/few_shot_k` 需**同时改 config 白名单和 render_config**（想要 radio 得改渲染，不只加键）。

**接线点**：`config.py`（+键 +渲染）；`analyze.py`（`_few_shot_examples` 注入 + rule 降级）；`store.py::training_queue`（新）；`run.py`（按 `mode` gate `push_report_notice`、注入分类器）；cron 配置（文档说明，非代码）。

### Phase D · 半自动种子增长

**目标**：从数据/标注挖"判别性强且产品相关"的候选词，**分流**：产品词→建议进 watch 种子（扩召回），立场词→建议进关键词库（提精度），全部人工一键确认。

**候选抽取** `analytics.extract_seed_candidates`：`suggest_targets`（现产文档簇）→ 簇内正文抽 2-4 字 n-gram（无分词器，与现有字符匹配一致）→ 打分：
```
lift = (簇内文档频率) / (全语料文档频率)   # ★串味核心闸门：泛词到处出现 lift≈1 被拦；产品专名 lift 高放行
score = 0.4·簇内频率 + 0.3·min(1,lift/5) + 0.3·语义相似
丢弃：单字 / ∈STOPWORDS∪km.get_by_tag('stopword') / df<2 / lift<3.0 / score<0.5
```

**分流**：命中立场词集(complaint/selling_point/competitor/crisis_boost) → `kind=feature`；高 lift 专名样 → `kind=seed`；**不确定默认 feature**（更安全，绝不无把握扩召回）。

**落库/确认**：两路都 `km.add_suggestion`；feature 走 `/keywords` approve；**seed 走 `/watch` 新增"种子建议区" → `POST /api/seed{approve}` → `append_alias`**：
- `append_alias(entity, word)`：读 watch.yaml 原文 → **surgical 文本插入**（保注释/must_not，不走 yaml.dump 抹注释）→ `_validate_watch` 校验 → `.bak` 备份 → 写盘（复用 `/api/watch` 同套安全路径）；surgical 未匹配(block-style)→降级预填 `/watch` 编辑器人工保存。
- `approve_suggestion` 加 guard 拒 `seed_alias`（防误点把种子塞进 keywords 表）。

**接线点**：`analytics.py`（+`extract_seed_candidates/mine_and_queue/append_alias/STOPWORDS`）；`keywords.py`（approve 加 guard，复用 add/reject）；`dashboard.py`（render_watch 加种子区、POST `/api/seed` 过 `_write_allowed`+`_validate_watch`）；`cli.py suggest` 加 `--write`。

### Phase B · 词-标签知识图谱（最后、可选）

> ★评审：**这是最强 YAGNI 目标**——与 Phase C 从同一张标注表学习，却只产"软提示"。
> **仅当 C 落地后被证明不够用（如立场判定系统性偏差、需可解释性）才建**，且优先做成**静态表而非力导图**。

**最小可交付（90% 洞察 / 5% 代码）**：新表 `graph_edges(src,dst,rel,weight=NPMI,cooccur,entity_id)`，
从 annotations 的圈词×标签共现算 **NPMI**（`MIN_SUPPORT` 起步设 2，防冷启动长期空），
出一张 **"词 → 最强共现标签"静态表**（如 `退款 → 立场:投诉 NPMI.62/18帖`）。消费三处：
给 `rule_extract` 破平先验（仅规则模糊时用，绝不硬覆盖）、给 few-shot 注入"仅供参考"共现块、给种子挑词排序。
力导图 `/graph` 页是**可选升级**，非 MVP。全量重建（标注小表够快），无 annotations 时降级空图不崩。

## 5. 关键纠偏汇总（评审确认的真问题）

- **A. 枚举串味**：SUBJECTS/STANCES 必须在 Phase A 冻结为跨组件唯一常量，全仓统一写法。★阻塞级。
- **B. 采样↔范例冲突**：主动学习偏负面 → 正面立场档凑不齐样本 → 队列留 ~30% 给立场覆盖。★阻塞级。
- **C. 重要性口径**：用 `score.mention_equiv`（覆盖全极性）**不用 `risk`**（对非负面恒 0，会把高传播正面误判"低"）。
- **D. `day` 作用域**：Phase C 分类调用必须在 `if engine!="rule"` 块内，否则 NameError 打崩分析。
- **E. `holdout` 陷阱**：生产 `build_exemplars(holdout=False)`，否则白扔 20% 标注。
- **F. config 开关**：加键须同改白名单 + render_config（radio 要改渲染）。
- **G. 冷启动主体维**：官方/准官方/媒体离线无数据源 → 决策⑥**建 `account_registry` 白名单**做确定性判定（§4-Account），白名单未命中再 LLM/followers 兜底。"纯传播"离线仍难识别，早期如实标注。
- **H. 图谱降级 YAGNI**：Phase B 降为最后+可选+静态表。
